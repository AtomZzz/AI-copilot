## 核心认知

> **AI 应用 ≠ 传统 CRUD 应用。**
> 
> 传统 Java 后端的核心是：请求 → 业务逻辑 → 数据库 → 响应。
> 而 AI 应用的核心是：请求 → Context 构建 → LLM 推理（可能多轮） → Tool 调度（可能多次） → 状态流转 → 响应。

这意味着：**系统的不确定性大幅增加，状态管理复杂度显著提升，可观测性要求更高。**

### 为什么这样分层？

| 层次 | 职责 | 传统后端对比 |
|------|------|-------------|
| **Gateway/BFF** | 认证、限流、审计 | 和传统一样，但增加了 AI 会话管理 |
| **AI Application** | Agent 编排、RAG、Workflow | **AI 特有层**，传统后端没有 |
| **AI Runtime** | Prompt、Context、Memory、Model 调度 | **AI 特有层**，这是整个系统的大脑 |
| **Tool Runtime** | 工具注册、调度、权限、结果适配 | **AI 特有层**，连接 LLM 与业务系统 |
| **Business Service** | 具体业务逻辑 | 和传统后端一样 |
| **Infrastructure** | 存储、缓存、向量库 | 多了 pgvector |

**核心差异**：中间三层（AI Application、AI Runtime、Tool Runtime）是 AI 应用独有的。这也是从 Java 后端转型 AI 工程师最需要深入理解的部分。

## 二、AI Runtime 架构（系统大脑）

AI Runtime 是整个系统最核心的模块。它决定了：用户的一句话，如何变成一次完整的 AI 推理过程。

### 数据流视角

```
用户输入 "帮我查一下我的积分余额"
    │
    ▼
[1] Context Builder
    │  → 组装 System Prompt
    │  → 注入用户身份（employeeId, role, department）
    │  → 注入 Memory（最近对话历史）
    │  → 注入 RAG Context（如果相关）
    │  → 注入可用 Tool 列表（根据权限过滤）
    │
    ▼
[2] Prompt Assembler
    │  → System Prompt + Tools Schema + Memory + User Message
    │  → Token 预估 & 截断
    │
    ▼
[3] Model Router
    │  → 根据任务类型选择模型
    │  → Fallback 策略（主模型不可用时切换）
    │
    ▼
[4] LLM 推理
    │  → Streaming 输出
    │  → 如果 LLM 返回 Tool Call → 进入 Tool 调度循环
    │  → 如果 LLM 返回文本 → 直接输出
    │
    ▼
[5] Response / Tool Loop
    → 可能多轮：LLM → Tool → LLM → Tool → ... → Final Response
```

### 关键设计决策

**为什么 AI Runtime 独立为一层？**

在传统 Java 后端，Controller → Service → DAO 是确定性的调用链。但在 AI 应用中：
- LLM 可能决定调用 0 个、1 个或多个 Tool
- 可能触发多轮推理循环（ReAct）
- 每次推理的 Token 消耗不同
- 每次推理的延迟不同
- 可能触发 Guard Rails 拦截

这些**不确定性**要求我们把"AI 推理编排"独立出来，而不是散落在业务代码中。

## 三、Tool Runtime 架构（连接 AI 与业务）

Tool Runtime 是 AI 应用中最容易被低估、但在企业级场景中最重要的模块。

### 架构设计

```
┌──────────────────────────────────────────┐
│              Tool Registry                │
│  ┌──────────────────────────────────────┐ │
│  │  ToolName: query_points_balance      │ │
│  │  Description: 查询员工积分余额         │ │
│  │  Parameters: { employeeId: String }   │ │
│  │  RequiredPermissions: [points:read]   │ │
│  │  Category: HR                         │ │
│  │  TimeoutMs: 3000                      │ │
│  │  RetryPolicy: 2 times                │ │
│  └──────────────────────────────────────┘ │
│  ┌──────────────────────────────────────┐ │
│  │  ToolName: submit_leave_request      │ │
│  │  Description: 提交假期申请            │ │
│  │  Parameters: { type, startDate, ... } │ │
│  │  RequiredPermissions: [leave:write]   │ │
│  │  Category: Leave                      │ │
│  │  Confirmation: REQUIRED (写操作)       │ │
│  └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────┐
│            Tool Dispatcher                │
│  1. 权限校验（当前用户是否有权调用此 Tool）   │
│  2. 参数校验（LLM 生成的参数是否合法）        │
│  3. 审计记录（谁、什么时候、调了什么 Tool）    │
│  4. 执行（带超时控制）                       │
│  5. 结果适配（转成 LLM 可理解的格式返回）      │
│  6. 异常处理（Tool 失败时的降级策略）          │
└──────────────────────────────────────────┘
```

### 为什么 Tool Runtime 如此重要？

**传统后端思维**：你调用一个 Service 方法，参数是你写的，权限是你控制的。

**AI 工程思维**：LLM 决定调用哪个 Tool，LLM 决定传什么参数。这意味着：
1. **参数不可信** — LLM 可能编造不存在的 employeeId
2. **权限必须后置校验** — 不能因为 LLM 调了某个 Tool 就默认有权限
3. **写操作必须确认** — LLM 不能直接帮你提交假期申请，必须让用户确认
4. **结果必须适配** — Tool 返回的是 Java 对象，但要转成 LLM 能理解的文本/结构化数据

### 读操作 vs 写操作

这是一个关键分类：

| 类型 | 示例 | 策略 |
|------|------|------|
| **读操作（Read Tool）** | 查积分、查历史、查制度 | LLM 可直接调用，结果直接返回 |
| **写操作（Write Tool）** | 提交假期、提交 OT、填工时 | 需要 **Human-in-the-Loop**，先展示给用户确认 |

## 四、RAG 架构（企业知识问答）

### 文档入库流程（离线）

```
PDF/Word/Confluence → 文档解析 → 智能切片 → Embedding → pgvector
```

### 查询流程（在线）

```
用户提问
    │
    ▼
[1] Query Rewriting（优化查询，提升召回率）
    │
    ▼
[2] Retrieval（从 pgvector 检索 Top-K 相关片段）
    │
    ▼
[3] Rerank（对检索结果重排序，提升精确度）
    │
    ▼
[4] Context Injection（将相关片段注入 Prompt）
    │
    ▼
[5] LLM 生成回答（带引用来源）
```

### 设计要点

- **切片策略**：银行制度文档结构复杂（章节、条款、表格），不能用简单的固定长度切片。需要 **Semantic Chunking** + **结构感知切片**。
- **Rerank**：银行场景对精确度要求极高（制度条文不能答错），必须加 Rerank。
- **引用溯源**：回答必须标注来源文档和段落，银行审计需要。
- **知识更新**：制度更新后需要有版本管理和增量更新机制。

## 五、Memory 架构（对话记忆）

```
┌─────────────────────────────────────────┐
│              Memory Manager              │
│                                          │
│  ┌─────────────┐  ┌──────────────────┐   │
│  │ Short-term  │  │   Long-term      │   │
│  │ Memory      │  │   Memory         │   │
│  │ (会话内)     │  │   (跨会话)        │   │
│  └─────────────┘  └──────────────────┘   │
│                                          │
│  Short-term:                             │
│  - 当前会话的 Message History             │
│  - Sliding Window / Summary 策略          │
│  - 存储在 Redis（TTL 自动过期）            │
│                                          │
│  Long-term:                              │
│  - 用户偏好、常用查询模式                   │
│  - 存储在 PostgreSQL                      │
└─────────────────────────────────────────┘
```

### 为什么 Memory 如此关键？

传统后端是**无状态**的（每次请求独立）。但 AI 对话是**有状态**的：
- 用户说"帮我请假"，下一句说"下周一"，再下一句说"年假" — 这三句话是一个上下文
- 但 Token 有限，不能无限塞历史消息
- 需要 **Sliding Window**（保留最近 N 轮）或 **Summary**（压缩历史为摘要）策略

## 六、Workflow 架构（流程编排）

后期使用 LangGraph4j 实现，核心思想：

```
StateGraph:

  START → [Intent Router]
              │
              ├──→ [Points Agent]   → END
              ├──→ [Leave Agent]    → [Confirmation] → END
              ├──→ [RAG Agent]      → END
              ├──→ [Timesheet Agent]→ END
              └──→ [Fallback]       → END
```

每个 Agent 是一个**有状态的节点**，State 在节点之间流转。条件边（Conditional Edge）决定下一步去哪里。

这比传统的 if-else 路由强大得多，因为：
- 支持循环（Agent 可以多次调用 Tool）
- 支持分支（根据 State 动态决定路径）
- 支持断点恢复（Checkpoint，审批流中断后恢复）

**为什么 Prompt 需要版本管理？**

在传统后端，你改 SQL 需要测试和版本控制。Prompt 也一样 — 它是 AI 应用的"业务逻辑"。改了 Prompt 可能影响回答质量、Tool 调用准确率、Token 消耗。必须像管理代码一样管理 Prompt。

## 八、权限与安全设计

银行场景，安全是**第一优先级**。

```
┌─────────────────────────────────────────────┐
│              AI Safety Layer                 │
│                                              │
│  [1] Input Guard Rails                       │
│      - 敏感信息过滤（卡号、身份证、密码）        │
│      - Prompt Injection 检测                  │
│      - 意图边界检查（只处理业务相关请求）         │
│                                              │
│  [2] Authorization                           │
│      - 基于角色的 Tool 权限（RBAC）            │
│      - 数据行级权限（只能查自己的积分）           │
│      - 写操作二次确认                          │
│                                              │
│  [3] Output Guard Rails                      │
│      - 敏感信息脱敏（回答中不能包含他人隐私）      │
│      - 合规检查                               │
│      - 幻觉检测（关键业务回答需要置信度评估）      │
│                                              │
│  [4] Audit Trail                             │
│      - 每次对话完整记录                         │
│      - 每次 Tool 调用记录                      │
│      - 每次 LLM 推理记录（含 Token 消耗）        │
│      - 异常事件记录                             │
└─────────────────────────────────────────────┘
```

## 十、技术权衡总结

| 设计点 | 选择 | 原因 | 风险 | 演进方向 |
|--------|------|------|------|----------|
| Agent 框架 | Spring AI + LangChain4j | Java 生态最成熟 | LangChain4j 迭代快，API 可能变 | 后期引入 LangGraph4j |
| 向量库 | pgvector | 与 PostgreSQL 同库，运维简单 | 超大规模性能不如专用向量库 | 如有需要可迁移 Milvus |
| 对话存储 | Redis (短期) + PG (长期) | 热数据快，冷数据持久化 | Redis 宕机丢失会话 | Redis Sentinel / Cluster |
| 流式输出 | SSE | 简单、HTTP 友好 | 单向通信 | 如需双向可加 WebSocket |
| 工具权限 | RBAC + Tool 级注解 | 银行合规要求 | 权限模型复杂度高 | 细粒度 ABAC |
