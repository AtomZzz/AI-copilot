## 问题一：项目初始化 — Spring Initializr vs 手动搭建？

### 方案对比

#### 方案 A：Spring Initializr（推荐）

**优点：**
1. **标准化工程结构** — 自动创建标准的 Maven/Gradle 项目结构
2. **依赖管理自动化** — 自动生成 `pom.xml` 或 `build.gradle`，包含版本管理
3. **快速启动** — 5 分钟内完成基础框架搭建
4. **IDEA U 集成良好** — IDEA Ultimate 内置 Spring Initializr，体验流畅
5. **社区最佳实践** — 生成的结构符合 Spring Boot 官方推荐

**缺点：**
1. 需要手动添加一些 AI 相关依赖（Spring AI、LangChain4j 等可能不在默认模板中）
2. 部分配置需要后续调整

**适用场景：** ✅ **强烈推荐用于本项目**

---

#### 方案 B：手动搭建

**优点：**
1. 完全可控，从零开始理解每个配置项
2. 适合学习 Spring Boot 底层原理

**缺点：**
1. **耗时长** — 至少需要 30-60 分钟搭建基础框架
2. **容易出错** — 依赖版本冲突、目录结构不规范等问题频发
3. **重复劳动** — 大部分企业项目都用 Spring Initializr，手动搭建没有实际价值
4. **偏离重点** — 我们的重点是 AI 工程，不是 Spring Boot 脚手架

**适用场景：**  不推荐（除非你想深入学习 Spring Boot 内部机制）

---

### Tech Lead 建议

**使用 Spring Initializr + IDEA U 内置功能。**

理由：
1. 你是多年 Java 后端工程师，Spring Boot 项目初始化应该是"肌肉记忆"级别的操作
2. 本项目的核心价值在于 **AI Runtime 设计和实现**，而不是项目骨架搭建
3. Spring Initializr 生成的标准结构，后续团队成员接手时更容易理解
4. 我们可以在生成后，根据 AI 应用需求调整包结构和依赖

### 具体操作步骤

```
1. IDEA U → File → New → Project
2. 选择 "Spring Initializr"
3. 配置：
   - Project: Maven
   - Language: Java
   - Packaging: Jar
   - Java Version: 21
   - Spring Boot: 3.3.x (最新稳定版)
   
4. Dependencies（初始选择）：
   - Spring Web
   - Spring Data JPA
   - PostgreSQL Driver
   - Redis
   - Lombok
   - Validation
   
5. 生成后，我们再手动添加：
   - Spring AI (BOM)
   - LangChain4j
   - pgvector 驱动
   - SSE Streaming 支持
```

## 问题二：LLM 接入 — 国内大模型 vs OpenAI vs Ollama 本地模型？

### 先回答核心问题

**你的截图显示的是阿里云百炼平台，提供了丰富的通义千问（Qwen）系列模型。**

从截图中可以看到你有以下可用模型（免费额度充足）：
- qwen3.7-plus
- qwen-math-turbo
- qwen-vl 系列（多模态）
- qwen-plus / qwen-max
- deepseek-r1-distill-qwen-7b
- glm-5（智谱 GLM）

### 方案对比分析

#### 方案 A：阿里云百炼（通义千问 Qwen）✅ **推荐首选**

**优点：**
1. ✅ **网络稳定** — 国内访问，延迟低，不会被墙
2. ✅ **免费额度充足** — 截图显示每个模型都有 1,000,000 Token 免费额度
3. ✅ **性能优秀** — Qwen-Max 和 Qwen-Plus 在中文理解和 Tool Calling 方面表现优异
4. ✅ **合规性好** — 银行内部系统使用国内云服务更符合监管要求
5. ✅ **调试方便** — 频繁调用不用担心国际网络波动

**缺点：**
1. ️ Spring AI 对通义千问的支持需要确认（见下文"框架兼容性"部分）
2. ⚠️ 长期成本需要考虑（免费额度用完后需付费）

**适用场景：** ✅ **强烈推荐作为主要开发环境**

---

#### 方案 B：Ollama 本地模型

**优点：**
1. ✅ **完全免费** — 无 Token 费用
2. ✅ **数据隐私** — 所有数据在本地，适合敏感场景
3. ✅ **无限调试** — 可以随意测试，不用担心配额

**缺点：**
1.  **性能有限** — 本地模型的推理能力远不如云端大模型
2.  **Tool Calling 能力弱** — 大多数开源模型的 Tool Calling 准确率较低
3.  **资源消耗大** — 需要 GPU 或大量内存
4. ❌ **不稳定** — 小模型容易产生幻觉、格式错误

**适用场景：** ⚠️ **仅适合学习和原型验证，不适合企业级开发**

---

#### 方案 C：OpenAI / Azure OpenAI

**优点：**
1. ✅ **生态最成熟** — Spring AI、LangChain4j 原生支持最好
2. ✅ **Tool Calling 能力强** — GPT-4o 的 Tool Calling 准确率行业领先
3. ✅ **文档完善** — 遇到问题容易找到解决方案

**缺点：**
1. ❌ **网络问题** — 国内访问不稳定，需要代理
2. ❌ **合规风险** — 银行内部系统使用国外云服务可能有监管问题
3. ❌ **成本高** — GPT-4o 的 Token 费用较高

**适用场景：** ❌ **不建议作为银行内部项目的主要选择**

---

### Tech Lead 建议

**采用"双模型策略"：**

1. **主要开发环境：阿里云百炼（通义千问 Qwen）**
   - 利用充足的免费额度进行日常开发和调试
   - 网络稳定，符合银行合规要求
   - Qwen-Max 的 Tool Calling 能力已经足够支撑第二阶段的最小闭环

2. **备选/对比环境：Ollama 本地模型（可选）**
   - 用于概念验证和低成本测试
   - 不作为生产环境的依赖

3. **后期演进：Model Gateway 抽象**
   - 第五阶段引入 Model Gateway 后，可以轻松切换模型供应商
   - 如果未来发现 Qwen 在某些场景下不足，可以切换到其他模型

## 问题三：框架兼容性 — Spring AI / LangChain4j 是否支持国内大模型？

### 核心结论

**Spring AI 和 LangChain4j 都支持通义千问（Qwen），但支持程度不同。**

### Spring AI 对通义千问的支持

#### 现状（截至 2026 年初）

Spring AI 官方支持的模型包括：
- ✅ OpenAI (GPT 系列)
- ✅ Azure OpenAI
- ✅ Anthropic (Claude)
- ✅ Google Vertex AI (Gemini)
- ✅ Alibaba Cloud Tongyi Qianwen (通义千问) ← **已支持**
- ✅ Amazon Bedrock
- ✅ Mistral AI
- ✅ Ollama

**通义千问支持情况：**
- Spring AI 1.0+ 版本已经内置了 `AlibabaAiChatModel`
- 支持 Chat Completion API
- 支持 Streaming
- 支持 Function Calling（即 Tool Calling）

**参考文档：**
- Spring AI 官方文档：https://spring.io/projects/spring-ai
- Alibaba AI 模块：`spring-ai-alibaba-starter`

#### 使用方式

```xml
<!-- pom.xml -->
<dependency>
    <groupId>com.alibaba.cloud.ai</groupId>
    <artifactId>spring-ai-alibaba-starter</artifactId>
    <version>${spring-ai.version}</version>
</dependency>
```

```yaml
# application.yml
spring:
  ai:
    alibaba:
      api-key: ${DASHSCOPE_API_KEY}
      chat:
        options:
          model: qwen-max  # 或 qwen-plus
```

---

### LangChain4j 对通义千问的支持

#### 现状

LangChain4j 通过社区贡献者提供了通义千问支持：

- ✅ 有 `langchain4j-dashscope` 模块
- ✅ 支持 Chat Completion
- ✅ 支持 Streaming
- ✅ 支持 Function Calling

**使用方式：**

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-dashscope</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

---

### Ollama 支持情况

#### Spring AI

- ✅ 完全支持 Ollama
- 使用 `spring-ai-ollama-starter`
- 适合本地开发和测试

#### LangChain4j

- ✅ 完全支持 Ollama
- 使用 `langchain4j-ollama` 模块

---

### 框架选择建议

#### 当前阶段（第二~四阶段）

**推荐使用 Spring AI + LangChain4j 混合方案：**

| 功能 | 推荐框架 | 原因 |
|------|----------|------|
| 基础 Chat / Tool Calling | **Spring AI** | Spring 生态集成更好，配置更简单 |
| Agent 编排 / ReAct | **LangChain4j** | Agent 相关 API 更丰富 |
| RAG | **LangChain4j** | Retriever、Embedding 工具链更完善 |
| Workflow（第四阶段） | **LangGraph4j** | 专门的状态图引擎 |

**为什么混合使用？**
- Spring AI 的优势在于与 Spring Boot 的深度集成（配置管理、依赖注入）
- LangChain4j 的优势在于 AI 专用功能（Agent、RAG、Memory）
- 两者可以共存，互相补充

#### 依赖配置示例

```xml
<!-- Spring AI BOM -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.ai</groupId>
            <artifactId>spring-ai-bom</artifactId>
            <version>1.0.0-M4</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- Spring AI Core + Alibaba -->
<dependencies>
    <dependency>
        <groupId>org.springframework.ai</groupId>
        <artifactId>spring-ai-starter-model-chat</artifactId>
    </dependency>
    <dependency>
        <groupId>com.alibaba.cloud.ai</groupId>
        <artifactId>spring-ai-alibaba-starter</artifactId>
    </dependency>
    
    <!-- LangChain4j Core -->
    <dependency>
        <groupId>dev.langchain4j</groupId>
        <artifactId>langchain4j</artifactId>
        <version>0.35.0</version>
    </dependency>
    <dependency>
        <groupId>dev.langchain4j</groupId>
        <artifactId>langchain4j-dashscope</artifactId>
        <version>0.35.0</version>
    </dependency>
    
    <!-- LangGraph4j（第四阶段引入） -->
    <dependency>
        <groupId>dev.langchain4j</groupId>
        <artifactId>langgraph4j</artifactId>
        <version>0.35.0</version>
    </dependency>
</dependencies>
```

## 常见问题预判

### Q1: 如果通义千问的 Tool Calling 效果不好怎么办？

**A:** 
1. 首先确认 Prompt 设计是否合理（Tool Description 是否清晰）
2. 尝试切换到 Qwen-Max（比 Qwen-Plus 更强）
3. 如果仍然不理想，可以考虑：
   - 使用 Structured Output 强制模型返回 JSON
   - 在后端做参数补全和校验
   - 第五阶段引入 Model Fallback 机制

### Q2: Spring AI 和 LangChain4j 会不会有冲突？

**A:** 
不会。它们是两个独立的库，可以共存。我们只会用各自的强项部分，不会产生依赖冲突。

### Q3: 如果我想先用 Ollama 本地模型试试呢？

**A:** 
完全可以。你可以：
1. 先用 Ollama + qwen2.5:7b 跑通最小闭环
2. 验证架构设计是否正确
3. 再切换到阿里云百炼进行正式开发

但需要注意：本地模型的 Tool Calling 准确率可能较低，可能需要更多调试。

### Q4: 银行内部系统用云服务是否合规？

**A:** 
这是一个好问题。通常：
- 阿里云是国内云服务提供商，符合中国监管要求
- 银行内部系统一般会使用私有化部署或专属云
- 如果是真正的生产环境，建议咨询银行的合规部门
- 但对于学习和原型开发，阿里云百炼是完全可行的
